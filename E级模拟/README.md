# E级生成/汇总时间模拟（22,680 节点）

模拟"体系生成"与"能量汇总"两个求解器外流程在 **22,680 节点**（8000 原子/节点，共 181,439,999 原子、n_c=60,479,999 碳）规模下的时间。生成无法真跑 1.8 亿原子（内存/IO 不允许），故**实测小规模 + 拟合标度律 + 外推**；汇总用编造数据实测。

## 生成（h_to_c 映射，生成的算法瓶颈）

`gen_projection.py` 实测旧法（brute-force argmin）与新法（scipy.cKDTree）的 h_to_c，拟合后外推：

| n_c（碳数） | 原子数 | 旧 brute (s) | 新 cKDTree (s) |
|---|---|---|---|
| 2,000 | 6,002 | 0.120 | 0.0013 |
| 4,000 | 12,002 | 0.456 | 0.0027 |
| 8,000 | 24,002 | 1.844 | 0.0053 |
| 16,000 | 48,002 | 6.928 | 0.0107 |
| 32,000 | 96,002 | — | 0.0221 |
| 100,000 | 300,002 | — | 0.0767 |

**外推到 22,680 节点（n_c=60,479,999）：**

| | 旧 brute O(N²) | 新 cKDTree O(N log N) |
|---|---|---|
| h_to_c 时间 | **1,209 天（≈3.3 年）** | **80.5 秒（≈1.3 分钟）** |
| 提速 | — | **≈ 1,298,995 ×** |

结论：旧法在 E 级完全不可行（年级）；新法把这一步压到分钟级。h_to_c 不再是生成墙。

> 注：80.5 s 仅为 h_to_c 这一步。完整生成还含 `gen.py` 碳骨架生成（Python 循环，60M 碳约分钟级）与写 22,680 个 block FDF（~15 GB 文本，IO 约分钟级）——这些是**中心化**生成的剩余成本，可通过"分布式按片段生成"进一步消除（见《生成优化文档》）。

## 汇总（combine_results，O(N) 求和）

`combine_sim.py` 用编造的 22,680 个 block / 22,679 个 cap / 22,679 个 dimer 能量实测 MFCC(1)+MBE(2) 求和：

- **汇总耗时：0.022 s**（O(N) 求和，~68k 个数）。

结论：汇总在 E 级也可忽略，不构成瓶颈。

## 复现

```bash
python3 gen_projection.py     # 生成 h_to_c 实测+外推（需 numpy/scipy）
python3 combine_sim.py        # 汇总实测（需 numpy）
```
