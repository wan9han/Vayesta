# 体系生成过程优化文档

## 1. 背景

弱扩展方案二（分块 + MFCC/MBE）的求解器部分在 1→16 节点已达成近 100% 弱扩展。但求解器之外还有两个串行流程：**体系生成**（生成聚乙烯链、切成 block/cap/dimer、写出 FDF）与**能量汇总**（combine_results，Σ blocks − Σ caps + …）。后者是 O(N) 求和，可忽略；前者在实测中"越来越慢"，本文件定位并解决该瓶颈。

## 2. 瓶颈定位：H→C 映射的 O(N²)

`weak_scale_pe.py:_build_fragments` 中，为把每个氢原子归属到它所键合的碳（进而归属到 block），原实现对每个 H 扫描**全部碳**取最近邻：

```python
cs = [碳原子下标，按坐标排序]
cpos = coords[cs]
h_to_c = {}
for h in h_idx:                              # 遍历每个 H
    d = np.linalg.norm(cpos - coords[h], axis=1)   # 与【所有】碳求距离
    h_to_c[h] = cs[int(np.argmin(d))]        # 取最近碳
```

复杂度为 **O(n_H × n_C) = O(N²)**（N 为总原子数）。这是体系生成的算法瓶颈：

- 16 节点（128,000 原子）：约 3.6×10⁹ 次元素运算，~10 秒，已与求解器同级；
- 22,680 节点（1.81×10⁸ 原子）：外推约 **1,209 天**（见 `E级模拟/`），完全不可行。

## 3. 优化：scipy.cKDTree，O(N log N)，结果不变

将"对每个 H 全表 argmin"换成对碳坐标建 **cKDTree**、对所有 H 一次性查询最近邻：

```python
from scipy.spatial import cKDTree
tree = cKDTree(cpos)                          # 建树 O(n_C log n_C)
_, nn = tree.query(coords[h_idx], k=1)        # 全部 H 的最近碳，O(n_H log n_C)
nn = np.atleast_1d(nn)
for h, j in zip(h_idx, nn):
    h_to_c[h] = cs[int(j)]
```

**正确性保证**：cKDTree 的 k=1 查询返回的就是欧氏距离最近碳的下标，与原 `argmin(norm)` **语义完全一致**；聚乙烯 C–H 键长 1.10 Å、最近邻碳唯一（无并列），故映射结果逐位相同。实测 n_c = 8 / 50 / 200 / 1000 / 4266 五档，旧法与新法的 `h_to_c` 字典**完全相等**（已断言验证）。

## 4. 性能对比（实测 + 外推）

`E级模拟/gen_projection.py` 实测：

| n_c | 原子数 | 旧 brute (s) | 新 cKDTree (s) |
|---|---|---|---|
| 2,000 | 6,002 | 0.120 | 0.0013 |
| 8,000 | 24,002 | 1.844 | 0.0053 |
| 16,000 | 48,002 | 6.928 | 0.0107 |
| 100,000 | 300,002 | — | 0.0767 |

外推到 22,680 节点（n_c≈6.05×10⁷）：旧法 **1,209 天** → 新法 **80.5 秒**，**提速约 130 万倍**。h_to_c 不再是生成墙。

## 5. 计时埋点

为便于持续监控，给两个流程加了计时输出：

- **生成**（`weak_scale_pe.py` main）：打印
  `[gen-time] chain=..s  fragments(h_to_c+build)=..s  write_inputs=..s  write_artifacts=..s  total=..s`
- **汇总**（生成的 `combine_results.py`）：打印
  `[combine-time] parse+sum = ..s  (N blocks, N-1 caps, N-1 dimers)`

## 6. 生成目录体积优化

E 级（22,680 节点）生成的输入目录理论约 65 GB（见 `E级模拟/README.md`），其中两项是明显浪费，已加开关治理：

**6.1 赝势共享（`--shared-pseudo`，默认开）。** 原实现把 C.psf/H.psf 拷进每一个 block/dimer/cap/full 目录（~260 KB/目录）。改为只写一份到 `_pseudos/`，其余目录用相对符号链接（SIESTA 经链接读取，行为不变）。实测 16×8000：拷贝 44 MB → 共享 **34 MB**（省 ~10 MB）；外推 22,680 节点省 **~14.5 GB**。`--no-shared-pseudo` 可恢复逐目录拷贝。

**6.2 跳过 full 基线（`--full-baseline`，默认开）。** "未分块整链"基线在 E 级 SIESTA 根本跑不动（正是方案二要回避的），其 FDF 单独约 13 GB。`--no-full-baseline` 在 E 级生成时跳过它。实测 16×8000：省 ~8 MB（full FDF 8.6 MB）；外推 22,680 节点省 **~13 GB**。

两者合计：22,680 节点目录 **~65 GB → ~38 GB**（剩下主要是 dimer ~25 GB + block ~13 GB，是真正要算的作业输入）。

**6.3 实测生成时间（本机，16×8000 = 128,000 原子）。**
```
[gen-time] chain=0.43s  fragments(h_to_c+build)=0.30s  write_inputs=0.69s  total=1.42s
```
墙时 1.50 s，峰值内存 129 MB。其中 h_to_c 仅 0.30 s（cKDTree 优化后），已不再是瓶颈。

## 7. 结论与后续

- 汇总为 O(N) 求和，22,680 节点实测 **0.022 s**，不构成瓶颈。
- 生成的 O(N²) 瓶颈（h_to_c）已由 cKDTree 消除，E 级从"不可行（年级）"降为"分钟级"，且结果与原实现完全一致。
- **剩余的中心化成本**：`gen.py` 碳骨架的 Python 循环（60M 碳约分钟级）与写 22,680 个 block FDF（~15 GB 文本，IO 约分钟级）仍为单点串行。进一步优化方向是**分布式按片段生成**——每个节点依据聚乙烯 zigzag 的确定性几何直接生成自己那一段（O(block size)/节点），使生成与求解具备同等弱扩展结构，彻底消除中心行长尾。此为后续工程项。
